## 現在ベクトルDBに保存しているもの

**pgvector (PostgreSQL)** + **Ollama (nomic-embed-text)** の構成です。

### 保存内容（768次元ベクトル）

| テーブル | 何をベクトル化しているか | 保存タイミング |
|---|---|---|
| **Article** | `タイトル + 分析サマリー` を結合したテキスト | 記事を3軸分析した直後（非同期） |
| **CompareResult** | 同上（媒体比較の各記事分） | 媒体比較セッション終了の3秒後 |

ベクトル以外に保存しているもの: タイトル・本文・URL・媒体名・経済/社会/外交スコア・感情トーン・要約・カウンターオピニオン など。

### 現在の利用箇所

| 機能 | エンドポイント | 内容 |
|---|---|---|
| **類似記事検索** | `POST /api/history/similar` | 検索テキストをベクトル化 → コサイン距離で最大5件を返す |

## ベクトルDBが向いている保存・処理

### ✅ 向いているもの

| ユースケース | 具体例 |
|---|---|
| **意味的類似検索** | 「防衛費」で検索 → 「軍事予算」「安保費」の記事もヒット（キーワード一致不要） |
| **クラスタリング・グルーピング** | 同一ニュースを複数媒体分まとめる（今の `groupArticlesByEvent` の Ollama 依存をベクトル距離で代替可能） |
| **重複除去** | 似た内容の記事を自動で弾く |
| **レコメンデーション** | 「この記事を読んだ人に似た記事を提示」 |
| **RAG（検索拡張生成）** | LLMに「過去の分析結果」を文脈として注入 |
| **異常検知** | 報道トーンが他媒体から大きく外れた記事を検出 |
| **多軸フィルタ + 類似検索の組み合わせ** | 「経済軸が保守よりかつ内容が財政政策に近い記事」を検索 |

### ❌ 向いていないもの

| ユースケース | 理由 |
|---|---|
| 完全一致・前方一致検索 | 通常のSQL `LIKE` / `=` の方が正確かつ高速 |
| 数値範囲フィルタ | `WHERE economic > 0.5` は普通のインデックスで十分 |
| 集計・統計 | `AVG(score)` などは SQL の得意領域 |
| リアルタイム更新が多いデータ | ベクトルインデックスの再構築コストがかかる |

## このプロジェクトで活かせる改善案

現状 `groupArticlesByEvent` は **Ollama LLM にグルーピングを任せている**ため、遅くてトークン消費も多いです。
→ **ベクトル距離（コサイン類似度 > 0.85 なら同一ニュース）** でグルーピングすると、LLM 呼び出しを省略でき、大幅に高速化できます。これが今の構成で最も効果的な改善になります。
