# 軽量分類・ルーティング戦略 — 【DONE】

> **実装状況**: Go バッチの `classify.go` にて、**Embedding 類似度分類 → LLM フォールバック → キーワードフォールバック** の 3 段カスケードが実装済み。

---

## 現状の分類アーキテクチャ

```
記事(title, summary)
  │
  ├─ 正常時 ─→ Ollama gemma3:4b (LLM) ──→ { category, subcategory, confidence }
  │             ~2-5秒/記事, バッチ時 ~10-30秒/20記事
  │
  └─ 障害時 ─→ キーワードマッチ (topic-classifier.ts) ──→ { category }
               <1ms, subcategory判定なし
```

**ボトルネック**: RSS更新時に20-50記事を分類 → gemma3:4bでも10-30秒かかる

---

## 戦略一覧

### 1. Embedding分類（推奨・最小コスト）

既に `nomic-embed-text` が稼働中。カテゴリ/サブカテゴリのラベル+説明文を事前にembedし、記事embeddingとのコサイン類似度で分類する。

```
[事前準備]
  34サブカテゴリの説明文 → nomic-embed-text → 34本の参照ベクトル（起動時1回）

[分類時]
  記事(title+summary) → nomic-embed-text → コサイン類似度 Top-1 → category/subcategory
```

| 項目 | 値 |
|------|-----|
| 追加モデル | **不要**（既存のnomic-embed-text） |
| 推論速度 | ~50-100ms/記事（embed + cosine） |
| バッチ | Ollama embed APIが一括対応済み |
| 精度 | カテゴリ8分類: 高い / サブカテゴリ34分類: やや曖昧な境界あり |
| 実装量 | 小（embeddings.ts + 類似度計算のみ） |

**利点**: 新しい依存なし。既存インフラで完結。LLM比50-100倍高速。
**弱点**: 「経済政策 vs 金融政策」のような意味的に近いサブカテゴリの境界が曖昧になりうる。

#### 精度改善オプション

- **複数例文方式**: 各サブカテゴリに5-10本の代表的記事タイトルをembedし、平均ベクトルを参照とする → 単一説明文より境界が明確に
- **confidence閾値**: 類似度が低い場合（< 0.5等）のみLLMにエスカレーション（後述のカスケード）

---

### 2. キーワード分類の強化（ゼロコスト）

既存の `topic-classifier.ts` はカテゴリのみ対応。これをサブカテゴリまで拡張する。

```typescript
// 現在: TOPICS[category].keywords → category
// 拡張: SUBCATEGORY_KEYWORDS[subcategory].keywords → { category, subcategory }
```

| 項目 | 値 |
|------|-----|
| 追加モデル | **不要** |
| 推論速度 | <1ms/記事 |
| 精度 | キーワード網羅度に依存。新語・文脈依存の分類に弱い |
| 実装量 | 中（34サブカテゴリ分のキーワード定義） |

**利点**: 最速。外部依存ゼロ。
**弱点**: メンテナンスコストが高い。文脈を読めない（「日銀の利上げ」→ 金融政策 vs 「利上げで住宅ローン負担増」→ 物価・消費）。

---

### 3. TF-IDF + ロジスティック回帰（従来ML）

記事テキストからTF-IDF特徴量を抽出し、軽量な分類器で判定する。

```
[学習フェーズ]
  LLM分類済み記事（DB蓄積） → TF-IDF → ロジスティック回帰 / SVM

[推論フェーズ]
  記事テキスト → TF-IDF → 分類器 → category/subcategory
```

| 項目 | 値 |
|------|-----|
| 追加依存 | Python (scikit-learn) or Rust/Wasm ライブラリ |
| 推論速度 | <5ms/記事 |
| 学習データ | LLM分類済みの記事が200-500本以上必要 |
| 精度 | 学習データの質に依存。十分なデータで90%+ |
| 実装量 | 大（Python環境 or Wasmビルド + モデルファイル管理） |

**利点**: 高速かつ高精度（データが十分なら）。
**弱点**: Node.js/Next.jsとの統合が面倒。学習データの蓄積フェーズが必要。

---

### 4. ONNX Runtime + 小型Transformer

日本語対応の小型BERTモデル（例: `cl-tohoku/bert-base-japanese`）をONNX化し、分類ヘッドを付けてfine-tuneする。

| 項目 | 値 |
|------|-----|
| 追加依存 | `onnxruntime-node` (~50MB) + モデルファイル (~400MB) |
| 推論速度 | ~20-50ms/記事 |
| 学習データ | 500本以上のラベル付き記事 |
| 精度 | embedding方式より高い境界判別能力 |
| 実装量 | 大（fine-tune環境 + ONNX変換 + Node.js統合） |

**利点**: LLMに近い精度でLLMの1/100のコスト。
**弱点**: fine-tune環境（GPU）が必要。モデル管理の運用負荷。個人開発には重い。

## カスケード（段階的エスカレーション）

上記の手法を組み合わせ、確信度に応じてエスカレーションする。

```
記事
 │
 ├─ Step 1: キーワードマッチ ─────→ ヒット＆一意 → 確定（<1ms）
 │                                    ↓ 未ヒット or 複数カテゴリ該当
 ├─ Step 2: Embedding類似度 ────→ similarity > 閾値 → 確定（~100ms）
 │                                    ↓ 閾値未満（曖昧な記事）
 └─ Step 3: LLM (gemma3:4b) ────→ 最終判定（~3秒）
```

**期待効果**:
- 典型的なニュース（災害速報、スポーツ結果等）の60-70%はStep 1で即確定
- Step 2でさらに20-25%を処理
- LLMが必要な曖昧な記事は全体の5-15%程度

→ **50記事あたりの所要時間: 30秒 → 3-5秒（推定）**

## 推奨プラン

### Phase A: Embedding分類の導入（即着手可能）

現在のインフラだけで実装可能。最もROIが高い。

1. 34サブカテゴリの参照embeddingを生成（`nomic-embed-text`で1回実行）
2. `classifyByEmbedding()` 関数を実装（cosine similarity → Top-1）
3. 既存の `classifyArticlesBatchLLM()` を置き換え
4. confidence閾値未満のみLLMにフォールバック（カスケード）

### Phase B: キーワード → Embedding → LLM の3段カスケード

Phase Aの結果を見て、曖昧な境界が多いサブカテゴリペアを特定。
そのペアに対してキーワードルールを追加し、Step 1の確定率を上げる。

### Phase C: 学習データ蓄積 → 従来ML（将来オプション）

DB蓄積が500記事を超えた時点でTF-IDF分類器の学習を検討。
Embeddingカスケードで十分な精度が出ていれば不要。
