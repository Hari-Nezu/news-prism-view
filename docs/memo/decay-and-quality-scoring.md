# 減衰・品質スコアリング設計

## 現状の問題

現在のランキングは **媒体数 → 記事数** の2軸のみ。以下が欠落している：

| 問題 | 症状 |
|------|------|
| 時間減衰なし | 3日前の大事件が最新ニュースより上に居座る |
| コピペ記事の水増し | 通信社配信の同一文面が複数媒体扱いでランク上昇 |
| セマンティック膨張 | 「経済」のような広いトピックが無関係な記事を吸収して巨大クラスタ化 |
| エンティティ変遷の無視 | 同一人物の別件ニュースが同じクラスタに混入 |

---

## 4つの減衰軸

### 1. 時間減衰（Time Decay） — 【DONE】

**目的**: 鮮度をランキングに反映する

**方式**: クラスタ内の最新記事の `publishedAt` から指数減衰

```
timeScore = exp(-λ · ageHours)
λ = ln(2) / halfLifeHours
halfLifeHours = 12
```

| 経過時間 | timeScore |
|---------|-----------|
| 0h | 1.00 |
| 6h | 0.71 |
| 12h | 0.50 |
| 24h | 0.25 |
| 48h | 0.06 |

**実装箇所**: `store.go` のランキングソート（実装済み）

**設計判断**:
- 半減期12時間は日本のニュースサイクル（朝刊→夕刊→翌朝刊）に合致
- `max(publishedAt)` を使用。クラスタ内の最新記事が基準
- 環境変数 `TIME_DECAY_HALF_LIFE_HOURS` で調整可能に

---

### 2. 情報密度（Copy Detection / uniqueSources） — 【一部DONE】

**目的**: 通信社配信のコピペ記事を検出し、見かけの媒体数水増しを防ぐ

**現在の実装**: `sourceCount` (uniqueSources) をカウントし、媒体数に応じた `sourceScore` (3.0, 2.0, 1.0) を `store.go` で計算。

**未実装**: 以下のコピペ類似度 (0.97) による独自報道カウント (uniqueVoices) のグラフ構築ロジックは未実装。

**方式**: クラスタ内の記事ペア間コサイン類似度で「独自記事」をカウント

```
1. クラスタ内の全記事ペアのコサイン類似度を計算
2. similarity > 0.97 のペアを「コピー関係」としてグラフ構築
3. 連結成分 = 1つの「独自報道」とみなす
4. uniqueVoices = 連結成分数
```

| 状況 | sourceCount | uniqueVoices |
|------|-------------|--------------|
| 5社が独自取材 | 5 | 5 |
| 3社独自 + 2社が共同配信コピペ | 5 | 4 |
| 5社とも共同通信コピペ | 5 | 1 |

**ランキングへの反映**:
- 現在の `sourceCount` を `uniqueVoices` で置き換え
- `coveredBy` は全媒体を維持（表示用）
- `uniqueVoices` をスナップショットに保存（透明性）

**実装箇所**: `store.go` の `uniqueSources` を拡張

**閾値の根拠**:
- 0.97 は同一文面のリード文改変（媒体名差し替え等）を許容
- 0.95 だと独自取材でも類似トピックが巻き込まれる
- 環境変数 `COPY_SIMILARITY_THRESHOLD` で調整可能に

---

### 3. セマンティック・ラジアス制限（Semantic Radius Cap）

**目的**: クラスタの意味的拡散を防ぎ、無関係な記事の吸収を阻止する

**補足**: 完全に同一のロジックではないが、`crossCategoryThresholdOffset` (0.08) によるカテゴリ・ソフトゲートが `group.go` に実装済みで、異カテゴリ間の誤混入を抑制している。

**方式**: クラスタ内の全記事とセントロイドの最小類似度を監視

```
radius = 1.0 - min(cosineSim(article, centroid) for article in cluster)
maxRadius = 0.20  （コサイン距離で0.20 = 類似度0.80以上を保証）
```

**挙動**:
- 新記事をクラスタに追加する前に、追加後の radius を予測
- `radius > maxRadius` なら追加を拒否（新規クラスタに配置）
- 既存の `threshold`（0.87）はクラスタへの参加条件、`maxRadius` はクラスタの拡散上限

```
[threshold=0.87]  記事がクラスタに入れるか？ → centroidとの類似度
[maxRadius=0.20]  クラスタがこれ以上広がれるか？ → 最も遠い記事との距離
```

**実装箇所**: `group.go` の `GroupArticles` ループ内

**なぜ threshold だけでは不十分か**:
- centroid は記事追加のたびに移動する
- 初期記事A（政策金利）→ 記事B（為替影響, sim=0.88）→ centroid移動 → 記事C（株式市場, sim=0.88 vs 新centroid）
- A-C間は sim=0.75 だが、各ステップでは閾値をクリアしている
- radius制限がこのドリフトを検出する

---

### 4. エンティティ変遷チェック（Entity Transition Guard）

**目的**: 同一人物・組織の異なるニュースイベントがクラスタに混入するのを防ぐ

**方式**: タイトルからエンティティ（人名・組織名）と動詞句を抽出し、イベント同一性を判定

```
1. クラスタのタイトル群から頻出エンティティを抽出（n-gram + 出現率50%以上）
2. 新記事のタイトルからエンティティを抽出
3. エンティティ重複あり かつ 動詞句が異なる → 別イベントの可能性
4. この場合、類似度に 0.85倍 のペナルティを適用
```

**例**:
- クラスタ: 「岸田首相 訪米」「岸田首相 バイデン大統領と会談」
- 候補: 「岸田首相 辞任表明」
- → エンティティ「岸田首相」一致、動詞句「訪米/会談」vs「辞任」不一致 → ペナルティ

**実装方式の選択肢**:

| 方式 | 精度 | コスト | 遅延 |
|------|------|--------|------|
| A: n-gramマッチ + 動詞辞書 | 中 | ゼロ | <1ms |
| B: LLMで判定 | 高 | API呼び出し | 数秒 |
| C: embeddingの差分方向分析 | 中-高 | ゼロ | <1ms |

**推奨: 方式A**（n-gramマッチ）
- パイプラインのgroupステージはCPUのみで完結すべき（LLM依存を増やさない）
- 日本語の動詞句は文末に集中するため、タイトル末尾の形態素で十分判別可能
- 方式Cは興味深いが、embedding空間で「同一人物の別イベント」と「同一イベントの別角度」を区別するのは困難

**動詞句辞書**（初期セット）:
```go
// タイトル末尾に出現する動作表現
actionPatterns = []string{
    "辞任", "就任", "逮捕", "死去", "訪問", "会談",
    "発表", "批判", "提案", "決定", "否定", "合意",
    "選出", "解散", "可決", "否決", "署名", "撤回",
}
```

## 統合スコア

```
finalScore = sourceScore × timeScore

sourceScore:
  - uniqueVoices >= 3  → 3.0（トップニュース級）
  - uniqueVoices == 2  → 2.0（複数社が独自取材）
  - uniqueVoices == 1  → 1.0（単独報道）
```

**ランキング順**: `finalScore DESC` → `uniqueVoices DESC` → `len(articles) DESC`

**注意**: スコアを連続値にしすぎると予測不能になる。離散的なティアの方がデバッグしやすく、ユーザーにも説明できる。

## 実装優先度

| 順序 | 軸 | 理由 | 影響範囲 |
|------|-----|------|---------|
| 1 | 時間減衰 | 最もユーザー体験に直結。実装も最小 | `store.go` |
| 2 | 情報密度 | コピペ問題は媒体数ランキングの信頼性を損なう | `store.go` |
| 3 | セマンティック・ラジアス | クラスタ品質の根本改善 | `group.go` |
| 4 | エンティティ変遷 | 精度は高いがエッジケースが多い。3まで入れてから評価 | `group.go` |

## 将来検討（未採用）

以下は4軸と直交するが、現時点では実装しない。具体的な不満が運用で出てから再検討する。

### 報道加速度（Velocity）
- 「古いが急展開中」のニュースを拾う軸。時間減衰の逆側の問題
- クラスタ内の直近N時間の記事追加レートで計測
- 課題: 毎時バッチの粒度では加速度の解像度が低い。バッチ間差分（前回スナップショットとの比較）で近似が必要で、実装コストが高い

### 話題飽和（Saturation）
- 数日間トップに居座る大事件を沈める軸。新記事が追加され続ける限り時間減衰では落ちない
- 過去N回のスナップショットで連続上位にいたクラスタにペナルティ
- 課題: 現在の「各バッチが独立」アーキテクチャにスナップショット間の状態比較を追加する必要がある
