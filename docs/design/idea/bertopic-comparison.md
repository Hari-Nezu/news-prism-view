# 現行クラスタリング vs BERTopic 比較

## 現行ロジック: Greedy Cosine Clustering + LLM Refine

### パイプライン

```
記事収集 → Embedding(ruri-v3) → 分類 → グルーピング → LLM精査 → 命名 → コンセンサス → 保存
```

### グルーピング (`group.go`)

**アルゴリズム:** 貪欲法コサイン類似度クラスタリング

1. 記事を順番に処理
2. 既存クラスタのセントロイドとコサイン類似度を計算
3. 閾値（~0.87）を超える最も類似するクラスタがあれば合流
4. なければ新規クラスタを作成
5. セントロイドは全記事の平均ベクトルで都度更新

**特性:**
- 次元削減なし — 768次元の埋め込みをそのまま比較
- クラスタ数は自動決定（閾値ベース）
- O(n*k) — 記事数 n, クラスタ数 k
- 処理順序に依存する（同じデータでも順序が違えば結果が変わる可能性）

### 精査 (`refine.go`)

1. **Triage:** クラスタ内min類似度 < intraThresh → split候補、クラスタ間centroid類似度 > interThresh → merge候補
2. **Critique:** LLMが記事タイトルを見て split/merge/move/coherent を判定（最大2ラウンド）
3. **Revise:** 判定に従い実際にクラスタを再構成

## BERTopic: Embeddings + UMAP + HDBSCAN

### パイプライン

```
文書 → Embedding → UMAP次元削減 → HDBSCAN密度ベースクラスタリング → c-TF-IDF トピック表現
```

### 各ステップ

| ステップ | 手法 | 役割 |
|:--|:--|:--|
| Embedding | 任意のモデル（Sentence-BERT等） | 文書をベクトル化 |
| UMAP | 非線形次元削減（768→5程度） | 高次元空間の局所構造を保持しつつ低次元化 |
| HDBSCAN | 密度ベース階層クラスタリング | 密度の高い領域をクラスタとして検出 |
| c-TF-IDF | クラスタ単位のTF-IDF | トピックのキーワード表現を生成 |

**特性:**
- ノイズ（どのクラスタにも属さない文書）を明示的に扱える
- クラスタ数は自動決定（密度ベース）
- 処理順序に非依存（決定論的ではないが、順序に起因する偏りはない）
- UMAP の確率的性質により実行ごとに結果が若干変動

---

## 比較

| 観点 | 現行 | BERTopic |
|:--|:--|:--|
| **次元削減** | なし（768次元直接比較） | UMAP（5次元程度へ） |
| **クラスタリング** | 貪欲法コサイン類似度 | HDBSCAN（密度ベース） |
| **クラスタ数決定** | 閾値ベース（自動） | 密度ベース（自動） |
| **ノイズ処理** | なし（全記事がどこかに属する） | あり（外れ値は -1 ラベル） |
| **順序依存性** | あり | なし |
| **トピック表現** | LLM命名 + kagome形態素キーワード | c-TF-IDF キーワード |
| **精査** | LLM Critique-Revise 2ラウンド | なし（後処理でトピック統合は可能） |
| **計算コスト** | 低（O(n*k) + LLM呼び出し） | UMAP・HDBSCANに依存（中〜高） |
| **実装言語** | Go | Python（scikit-learn依存） |
| **必要ライブラリ** | なし（自前実装） | umap-learn, hdbscan, scikit-learn |

## 現行アプローチの強み

1. **シンプルで透明** — 閾値1つでクラスタリングが理解・調整可能
2. **LLM精査による品質保証** — 統計的手法だけでは判断しにくい「トピックの一貫性」をLLMが補完
3. **Go単体で完結** — Python依存・追加ライブラリ不要
4. **ニュース記事の特性に合致** — 日次バッチで数百件規模ならgreedy法で十分な精度

## BERTopic の強み

1. **密度ベースのノイズ検出** — 無関係な記事を明示的に除外できる
2. **順序非依存** — 記事の入力順による結果のブレがない
3. **UMAP による次元の呪い回避** — 高次元空間ではコサイン類似度の識別力が低下する問題を緩和
4. **階層的トピック構造** — トピック間の親子関係を自然に表現可能
5. **c-TF-IDF** — LLM不要でトピックキーワードを抽出

## BERTopic 導入時の懸念

1. **Python依存の追加** — 現行GoパイプラインにPythonプロセスを組み込む必要がある
2. **UMAPの非決定性** — `random_state` 固定でも完全な再現性は保証されない
3. **小規模データでのHDBSCAN** — 日次バッチが数十件の場合、密度推定が不安定になる可能性
4. **チューニングの複雑さ** — UMAP（n_neighbors, min_dist, n_components）+ HDBSCAN（min_cluster_size, min_samples）のパラメータ空間が広い
5. **LLM精査との重複** — 現行のCritique-Reviseが既にクラスタ品質を担保しており、HDBSCAN導入の限界効用が問題

---

## 導入するなら

BERTopic の要素を段階的に取り入れるアプローチ:

1. **UMAP のみ導入** — 768次元→低次元へ削減してから現行 greedy clustering を適用。次元の呪い緩和効果を検証
2. **ノイズ検出の追加** — セントロイドとの類似度が一定以下の記事を「未分類」として分離（HDBSCAN不要で近い効果）
3. **HDBSCAN 置換** — greedy clustering を HDBSCAN に置き換え。LLM Refine は残す

Go で UMAP/HDBSCAN を実装するのは現実的ではないため、Python マイクロサービスとして分離するか、バッチ前処理として実行する構成になる。
