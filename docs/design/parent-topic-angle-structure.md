# 親トピック / 子トピック（angle）構造メモ（2026-04-09）

## 背景

現状のバッチでは、同一トピック判定されたグループの中に、明らかに異なるニュースが混入することがある。

一方で、実際には「完全に別ニュース」ではなく、

- 同じ大きな出来事圏を扱っている
- ただし論点や切り口が異なる

というケースも多い。

例:

- イラン情勢 > 外交安保
- イラン情勢 > 原油・エネルギー価格
- イラン情勢 > 日本政府対応
- イラン情勢 > 米国市場への影響

このため、単一クラスタに全部入れるのではなく、

- 親トピック: 同じ出来事圏・大テーマ
- 子トピック: その中の論点・切り口

の2層で表現する方が自然ではないか、という考えを整理する。

## 結論

方向性として妥当。

特に NewsPrism のように「同じ出来事を媒体ごとにどう報じたか」を見るプロダクトでは、
単一の group に全てを押し込むより、

- 同じ親トピックの中で
- どの angle を各媒体が拾っているか

を見せられる方が価値が高い。

ただし、いきなり全面導入するより、段階導入が必要。

## 解きたい問題

### 1. 本当に別ニュースなのに混ざっている

これは grouping 精度の問題。

例:

- 政治ニュースと経済ニュースが embedding の近さだけで混ざる

対策:

- まず category hard gate を入れて誤混入を減らす

### 2. 同じ大テーマだが論点が違う

これは grouping の粒度設計の問題。

例:

- 同じ「イラン情勢」でも、軍事報道と原油価格報道は別の角度

対策:

- 親トピックと子トピックを分けて持つ

## 提案する概念整理

### 親トピック

役割:

- 出来事圏・大きなテーマを表す
- ユーザーが「何の話か」を把握するための主見出し

例:

- イラン情勢
- 日米関税交渉
- 参院選
- 日銀政策修正

英語の内部名候補:

- `parentTopic`
- `eventTopic`
- `theme`

実装上は `parentTopic` が最も誤解が少ない。

### 子トピック

役割:

- 同じ親トピックの中の論点・焦点・切り口
- 現在の groupTitle に近い粒度

例:

- 外交安保
- 原油・エネルギー価格
- 日本政府の対応
- 市場への影響

英語の内部名候補:

- `angle`
- `focus`
- `subTopic`
- `lens`

実装上は `angle` か `focus` が扱いやすい。
個人的には `angle` が一番わかりやすい。

## 推奨する表現

UI 表示としては以下が自然。

- `イラン情勢 / 外交安保`
- `イラン情勢 / 原油・エネルギー価格`

または

- `イラン情勢 > 外交安保`
- `イラン情勢 > 原油・エネルギー価格`

文言として「マインドマップ」は発想としては近いが、
プロダクト仕様・データ設計の名前としてはやや曖昧。

実装用語としては次のどちらかがよい。

- `parent topic + angle`
- `theme + focus`

推奨:

- 画面表示: `親トピック > angle`
- 内部フィールド: `parentTopic`, `angle`

## 現行構造との対応

現状の主な概念:

- `category`
- `subcategory`
- `groupTitle`
- `topic`（整理予定）

これをそのまま置き換えるのではなく、役割を分けるべき。

### category

役割:

- 政治、経済、国際などの大分類
- grouping の安全装置

これは引き続き必要。

### subcategory

役割:

- taxonomy 上のより細かい分類
- angle 候補の補助信号

ただし `subcategory` 自体をそのままユーザー表示の angle にする必要はない。

### groupTitle

役割:

- 現状は「見出し」と「論点名」を兼ねている

親子2層を導入するなら、責務を分けたい。

候補:

- `parentTopic`: 大テーマ
- `angle`: 論点
- `groupTitle`: 表示用の最終ラベル

たとえば表示用に `parentTopic + angle` を連結して `groupTitle` を生成してもよい。

## 期待できる効果

### 1. 混入と論点違いを分けて扱える

今は両方とも「クラスタリング失敗」に見えるが、
親子2層にすると次のように分離できる。

- 別カテゴリが混ざる: 真の誤り
- 同じ親トピック内で angle が分かれる: 正常な分岐

### 2. CoverageMatrix との相性が良い

たとえば同じ親トピックに対して、

- A社は外交面だけ報道
- B社は原油価格面だけ報道
- C社は両方報道

のような差を可視化しやすい。

### 3. summary / compare の説明力が上がる

「同じ話題をどう報じたか」だけでなく、
「同じ話題のどの論点を各社が重視したか」まで出せる。

## 想定データモデル

段階導入の初期案:

- `category`: 安全装置用
- `subcategory`: taxonomy 用
- `parentTopic`: 大テーマ
- `angle`: 論点
- `groupTitle`: 表示名

表示名生成案:

```text
groupTitle = parentTopic + " > " + angle
```

ただし angle が親トピックとほぼ同義で冗長な場合は、
`groupTitle = parentTopic` にフォールバックしてよい。

## 生成方法の考え方

### 第1段階

まずは親子構造を「明示的な保存対象」にしない。

やること:

- category hard gate で誤混入を減らす
- 既存 groupTitle の品質を上げる

### 第2段階

親トピックと angle を命名段階で生成する。

案:

- cluster ごとに `angle` を作る
- 近い cluster 群を束ねる `parentTopic` を別途作る

ここでは LLM か taxonomy 補助が必要になる可能性が高い。

### 第3段階

snapshot に `parentTopic` / `angle` を保存する。

そうすると:

- summary
- coverage
- inspect
- compare

で同じ構造を再利用できる。

## 導入順序

### Step 1

先にカテゴリ制約付き clustering を入れる。

理由:

- 今のまま親子を作ろうとしても、入力 cluster 自体が不安定
- まず「明らかに違うものが混ざる」を止める必要がある

### Step 2

親トピック候補を UI 上または命名段階で試す。

例:

- 今の `groupTitle` を angle 相当に寄せる
- 近縁 group を束ねる親見出しを別に出す

### Step 3

データモデルとして正式採用する。

例:

- `snapshot_groups.parent_topic`
- `snapshot_groups.angle`

または

- 親トピック単位の別テーブル

## リスク

### 1. 親トピック生成が難しい

「イラン情勢」のような上位ラベルは、人間には自然だが自動生成はやや難しい。

特に embedding だけでは、

- 出来事そのもの
- 経済影響
- 外交反応

をどこまで同じ親に束ねるかが曖昧になりやすい。

### 2. taxonomy と役割が衝突する

`subcategory` が angle に近づきすぎると、

- taxonomy 上の分類
- UI 上の論点名

が混ざる恐れがある。

そのため、

- `subcategory`: 分類用
- `angle`: 表示・比較用

として責務を分けた方がよい。

### 3. UI が複雑になる

1階層より2階層の方が理解しやすい場合もあれば、
情報量が増えて煩雑に見える場合もある。

そのため最初は、

- 詳細表示だけ親子を出す
- 一覧では圧縮表示する

などの調整が必要。

## 現時点の判断

この方向性は採用候補としてかなり良い。

ただし、今すぐ「クラスタリング問題の直接解」として扱うのは危険。
先にやるべきは以下。

1. category hard gate で異カテゴリ混入を止める
2. Go 側分類精度を上げる
3. その上で親トピック / angle 構造を導入する

## 次に issue 化するなら

タイトル案:

- 親トピック / angle の2層構造でニュース群を表現できるようにする
- snapshot group に parentTopic / angle 構造を導入する
- CoverageMatrix と summary を親トピック / angle 表示に対応させる

概要案:

- 単一 groupTitle に大テーマと論点を押し込めるのをやめる
- `parentTopic` と `angle` を分離して保存・表示する
- 同一大テーマ内の論点差を UI で扱えるようにする
