# YouTube報道機関チャンネル横断比較 — 調査・実装計画

> **動機**: Netflix が報道領域に参入してくる前提（既に米国で Vox / 60 Minutes 系の配信買収検討が進行）で、既存の動画報道勢力（テレビ局・通信社の公式 YouTube）の「報じ方の差」を定量化できる土台を、**今のうちに**ローカル計算だけで作る。
>
> **方針**: クラウド LLM の従量課金は青天井になるため、推論はすべて **ローカル Ollama + pgvector** で完結させる。クラウドは「将来の法人プラン顧客が自分の API キーで叩く層」に限定。
>
> **スコープ**: メディア各社の **公式 YouTube チャンネル** に対象を絞り、ニュースイベント単位で「報じた / 報じていない / どう報じたか」を可視化する。テレビ本放送・違法アップ・個人切り抜きは対象外。
>
> 関連: [[monetization-plan]] / [[news-rights-research]]

---

## 0. 結論先出し

1. **取得手段は `YouTube Data API v3` のみ**（クォータ無料 10,000 unit/日で報道機関 30 ch 程度なら回る）。`yt-dlp` 等のダウンロードは ToS 違反になりうるので原則使わない。
2. **字幕は「YouTube が配信する公式自動字幕 / 投稿者字幕」を `timedtext` で取得**し、**スコアリング後に本文 DB を破棄**して `embedding + 3軸スコア + 要約(50字)` だけ残す（[[news-rights-research]] G1 の安全側運用）。
3. **「同じニュースイベント」の判定は、RSS 側の既存クラスタ（`SnapshotGroup`）に動画を埋め込みで紐付ける** ことで実装する。動画だけで独立クラスタリングしない＝既存資産を使い回す。
4. **マネタイズ単価が一番上がるのは「沈黙メディア（動画版）」レポート**。テキストRSSでの「報じない自由」可視化に、動画という第二軸を載せると差別化が二段になる。
5. **ローカル計算は MacBook 16GB クラスで日次 100 動画程度が上限の目安**。 e5-large 1024d 埋め込み + Llama 3.1 8B 量子化で字幕 5 分相当を 90 秒前後で処理できる試算。

## 1. 既存資産との接続

| 既存 | 本計画での扱い |
|---|---|
| `RssArticle` + 3 軸スコア + `SnapshotGroup` クラスタ | **そのままイベント主軸**として使う。動画はここに紐付ける |
| `YouTubeVideo` モデル（単発動画分析用） | スキーマを **チャンネル定期収集向け** に拡張（`channelId` をマスタ化、`event_group_id` 追加） |
| `batch/internal/pipeline/steps/{embed,classify,group}.go` | **再利用**。入力ソースを「RSS本文」から「字幕テキスト」に差し替えるだけ |
| `shared/llm/embed.go`（e5-large 1024d） | そのまま動画字幕にも適用。ベクトル空間を統一することで RSS↔動画の最近傍検索が可能 |
| `docs/monetization/news-rights-research.md` §3.1 | 動画取扱の **法的ガードレール** をそのまま流用 |

→ **新規追加が必要なのは「収集ステップ」と「RSS↔動画マッチ判定」だけ**。既存パイプラインに 2 ステップ足す構成で済む。

## 2. 法的に取れる範囲（動画特化）

[[news-rights-research]] §3.1 の表に **本計画用の運用ルール** を補足する。

| 行為 | 可否 | 本計画の運用 |
|---|---|---|
| YouTube Data API でメタデータ取得 | ◎ | 全件これで取得 |
| `timedtext` で字幕取得（公開字幕のみ） | ◎ | **公開設定の字幕に限定**。投稿者が非公開にしている場合は自動字幕も取得しない |
| 字幕 → DB 保存（解析目的） | ○ 30条の4 で可 | **スコアリング後 30 日以内に本文削除**、`embedding + 50字要約` のみ残す |
| 字幕 → ユーザー表示 | × | **絶対に出さない**。表示は「タイトル+公式iframe+自社生成スコア」のみ |
| サムネ画像の表示 | △ | **元 URL を `<img src>` 直リンク**（自前 CDN は使わない、[[news-rights-research]] G5） |
| 動画埋め込み | ◎ | 公式 iframe で確定。**`channelId` が公式チャンネルかを毎回検証** |
| 「報じていない」と表示 | ⚠️ | **「データ上、取得期間内に該当クラスタに紐付く動画が観測されなかった」** という事実表現に統一（[[news-rights-research]] D7） |
| 「報じた事実 + 公開日時 + 軸スコア」を法人に納品 | ○ | **事実 + 独自数値**なので OK。要約文の納品は **生成過程で原表現が残らないこと** を n-gram 検査 |
| 字幕本文を含むデータセット販売 | × | [[news-rights-research]] D6 と同じ。**売らない** |

### 公式チャンネル判定

「公式」を主観で決めない。以下の OR で機械判定する:

1. チャンネル `customUrl` が媒体ドメインの公式リストにある（例: `@nhk`, `@TBS_NEWS`）
2. チャンネルの `description` に媒体公式 URL（`nhk.or.jp` 等）が含まれる
3. 媒体公式サイトのフッターからリンクされている（半年に 1 回手動で照合）

→ `media_channels` テーブルに **ホワイトリスト方式** で持つ。**該当しないチャンネルは自動収集しない**（個人切り抜きを誤って取り込むと [[news-rights-research]] D5 の「違法アップの埋め込み幇助」リスクが出る）。

## 4. 「同じニュースイベント」の紐付け設計

新規クラスタリングをしないことが本設計の肝。**RSS 側の `SnapshotGroup` を主軸**にし、動画はそこに **後付け** する。

```
┌─────────────────┐  既存
│ SnapshotGroup   │ ← RSS記事クラスタ（イベント単位）
│  id, label,     │
│  centroid_emb   │
└────────┬────────┘
         │ best-cosine match
         │ (threshold τ)
┌────────┴────────────────┐ 新規
│ YouTubeVideoEventLink   │
│  video_id, group_id,    │
│  similarity, matched_at │
└─────────────────────────┘
         ↑
┌────────┴────────┐  既存拡張
│ YouTubeVideo    │ + event_group_id (nullable)
│  (字幕embedding)│ + channel_id FK to media_channels
└─────────────────┘
```

### マッチ判定アルゴリズム
1. 動画字幕を **タイトル + 概要 + 字幕先頭 2 分** に絞って e5-large で埋め込み（**先頭 2 分制約** = 報道動画は冒頭にリードが集中する経験則 + 計算コスト圧縮）
2. 同日 ±2 日の `SnapshotGroup.centroid_emb` と pgvector の `<=>` でコサイン距離トップ 3 を取得
3. **τ ≥ 0.72**（既存 RSS グルーピングの閾値より緩め。話題語彙が共通だが映像メディア特有の表現差を吸収）でリンク確定
4. どこにも紐付かない動画は `event_group_id = NULL` で「**未分類動画**」プールに残し、週次で人手レビュー or BERTopic で動画専用クラスタを生成

### 「沈黙」検出
あるイベント（`SnapshotGroup`）に対し、**`media_channels` ホワイトリスト全社のうちリンクが無いチャンネル一覧** を SQL で出す。これがそのまま「動画版・沈黙メディア」。

```sql
SELECT mc.channel_id, mc.display_name
FROM media_channels mc
WHERE NOT EXISTS (
  SELECT 1 FROM youtube_video_event_link l
  JOIN youtube_videos v ON v.video_id = l.video_id
  WHERE l.group_id = $1 AND v.channel_id = mc.channel_id
    AND v.published_at BETWEEN $2 AND $3
);
```

## 5. ローカル計算スタックとコスト試算

### 想定環境
- MacBook Pro M3 16GB（手元実機想定）
- Ollama（既存）: `intfloat/multilingual-e5-large`（1024d, 既存資産）+ `llama3.1:8b-instruct-q4_K_M`（要約・反論用）
- pgvector（既存）

### 1 動画あたりの処理コスト（実測オーダーの試算）
| 処理 | 時間 | 備考 |
|---|---|---|
| Data API メタデータ取得 | ~0.2s | クォータ消費のみ |
| 字幕 DL（5 分相当） | ~1s | timedtext 1 リクエスト |
| 字幕クリーンアップ + chunk化 | ~0.5s | Go 側 |
| e5-large 埋め込み（chunk 4個） | ~6s | 既存ベンチ準拠 |
| SnapshotGroup マッチ（pgvector） | ~0.1s | インデックス前提 |
| 3 軸スコア（Llama 3.1 8B q4） | ~60s | 50 トークン出力 |
| 50 字要約 | ~20s | 同上 |
| **合計** | **~90s** | |

→ **日次 100 動画 = 2.5 時間/日**。**15 ch で 1 ch 平均 5〜10 本/日**として 75〜150 本、夜間バッチ枠で消化可能。
→ クラウド LLM で同等を回すと **月 $200〜400 オーダー**。**ローカルなら電気代のみで実質 0 円**、青天井化しない。

### 「3軸スコアだけ高精度版を欲しい」ケース
LLM 推論は本数が増えると 8B では足りない場面が出る。その場合の段階的アップグレード:
1. **Llama 3.1 8B (q4)** → 日次 100 本まで
2. **Qwen 2.5 14B (q4)** → 16GB ギリギリ・速度半減
3. **Cloud に逃がす** → ユーザー側 API キー必須にして転嫁

→ 初期は 1 で固定。 2 への切替は「精度評価で 8B が `bias_warning` 誤検出 > 15%」を超えたら検討、という閾値で運用。

## 6. 可視化 UI（既存 `/ranking` に追加するビュー）

### 6.1 イベント詳細ページに「動画版マトリクス」ブロックを追加
```
NHK    🎥 12:14 「○○について政府は…」 [▶️ 埋め込み]  軸: E -0.2 S +0.1 D -0.4
TBS    🎥 18:45 「○○、与党内に異論」     [▶️ 埋め込み]  軸: E -0.1 S +0.3 D -0.2
産経   — （取得期間内に該当動画なし）
朝日   🎥 21:02 「○○、識者は批判」       [▶️ 埋め込み]  軸: E +0.4 S +0.6 D +0.5
日テレ — （取得期間内に該当動画なし）
…
```
- **「なし」の理由は明示しない**（推測表示は名誉毀損リスク [[news-rights-research]] D7）
- 動画は **公式 iframe 埋め込みのみ**、サムネを自前 CDN しない

### 6.2 媒体ごとの軸ヒートマップ（既存 `/ranking` の動画版タブ）
- 過去 30 日のイベント × 媒体マトリクスを **報道有無 (0/1)** で色塗り
- 数値は 3 軸スコアの中央値を別カラムで併記
- **「沈黙率」 = 報道無しイベント / 全イベント** を媒体ごとに計算 → 既存ランキングと並列展示

### 6.3 SEO 用「定点観測」ページ（マネタイズ起点）
- 「2026 年○月 ○○事件、動画報道していなかった媒体」**自動生成 note 風記事**
- 構造化データで `NewsArticle` ではなく `AnalysisNewsArticle` schema.org を当てる（自社解析であることを明示）
- これが集客の主軸。動画 + 沈黙という二段差別化で SEO は刺さりやすい

## 7. マネタイズ（[[monetization-plan]] への加算分）

| 階層 | 既存（RSS） | 動画レイヤ追加で起こること |
|---|---|---|
| A: 広告 | `/ranking` に AdSense | 動画タブが滞在時間を伸ばす → CPM が改善（記事だけより動画埋め込みのほうが回遊が出る）。**月 +5,000〜10,000 円** |
| B: Pro 個人 (980円) | 過去アーカイブ・キーワード監視 | **「特定チャンネルの軸変動アラート」**（NHK の社会軸が直近 30 日で +0.5 シフト等）を追加。Pro 価値が一段増す。**転換率 +20% 想定** |
| C: 法人 (9,800〜29,800円) | 報道トーン監視 | **PR / 広告代理店の購買決裁が通りやすくなる**（動画があるとクライアント説明資料に貼れる）。**法人単価 +1.5 倍**、29,800 → 49,800 円プランを上に作れる |
| D: データ販売 | スコア・URLのみ | **動画メタデータ + 軸スコア + 公開日時** のデータセットが新規プロダクト化。学術向け 5〜10 万円 / セット。**字幕本文は売らない**（[[news-rights-research]] D6） |

### 動画レイヤ単体の追加売上想定（[[monetization-plan]] §3 の 10 万円ラインに対する増分）
| 項目 | 月額増分 |
|---|---|
| 広告 CPM 改善 | +7,000 円 |
| Pro 転換率改善（+20 人想定） | +19,600 円 |
| 法人プラン上位化（1 社 → 49,800 円） | +20,000 円 |
| データセット（隔月 1 件・7 万円平均） | +35,000 円 |
| **合計増分** | **+約 8 万円** |

→ RSS 単体での 10 万円到達計画に対し、**動画レイヤが乗ると 15〜18 万円ラインが射程**。

### Netflix 参入時の戦略的価値
Netflix が独自報道を出してきた瞬間、「Netflix の報道は他社と何が違うのか」を答えられる **第三者観測データ** を最初に持っている事業者になれる。これは **メディア研究者・ジャーナリスト・国会議員秘書** にとってのキラー資料になる。Netflix 上陸後の数ヶ月だけでも **臨時レポート販売**（1 本 5〜10 万円）で売上立つ余地。

## 8. リスクと避けるべき罠

| リスク | 対応 |
|---|---|
| 公式チャンネル誤認定で個人切り抜きを取込 | §2 のホワイトリスト方式を厳守。`media_channels` への追加は手動 PR レビュー必須 |
| 字幕本文がユーザー画面に漏れる | API 層で `transcript` フィールドを response から **明示除外**（Zod スキーマで `.omit({transcript: true})`）。E2E テスト追加 |
| YouTube ToS の `timedtext` 取得が将来制限される | **既存 captions API への切替パス** を維持。`captions.list` + `captions.download` は OAuth 必須で動画オーナー権限なので、自動字幕は timedtext 経由で取得することを前提に **取得手段抽象化レイヤ** を 1 段挟む |
| 「沈黙メディア」表示の名誉毀損化 | 文言は **「観測期間内に該当クラスタの動画が確認できませんでした」** で統一。UI に「これは未報道を意味しません」注記を常時表示 |
| ローカル LLM の精度ドリフト | 既存 `compare-eval-models` スキル + `rss_*_eval` 評価枠を **動画字幕版に複製**（`yt_*_eval` テーブル）。週次で精度監視 |
| 3軸スコアの政治的反発 | [[monetization-plan]] §4 と同じく **採点ロジック公開** を盾にする。動画版は「冒頭 2 分のみで判定している」旨も明示 |
